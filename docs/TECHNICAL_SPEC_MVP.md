# Техническое задание: стабильный публичный MVP Telegram-бота для инкубации, кормов и учета яиц

Актуализация: документ начинался как MVP-ТЗ, часть пунктов уже реализована. Текущее состояние проекта отражено в `README.md`, `docs/ARCHITECTURE_NOTES.md` и `docs/ROADMAP.md`. Этот документ остается базовым продуктово-техническим ориентиром.

## 1. Назначение документа

Документ описывает требования к подготовке Telegram-бота к стабильному публичному MVP: запуску на сервере, надежной работе без ручного вмешательства, безопасному хранению данных, понятным пользовательским сценариям, базовой эксплуатации, администрированию и подготовке к дальнейшей монетизации.

## 2. Цель MVP

Сделать так, чтобы незнакомый пользователь мог самостоятельно пользоваться ботом без участия разработчика:

- создать и вести партию инкубации;
- получать корректные напоминания;
- видеть календарь действий и уход после вывода;
- учитывать корма, пополнения и расход;
- вести поголовье и стада;
- учитывать ежедневный сбор яиц и временно не несущихся кур;
- настраивать уведомления и базовые параметры хозяйства;
- безопасно возвращаться в меню и восстанавливаться после ошибок ввода.

Главный критерий MVP: бот стабильно работает на сервере, не теряет пользовательские данные, не отправляет дублирующиеся напоминания и имеет понятные сценарии для обычного пользователя.

## 3. Текущее состояние

Проект уже содержит:

- Telegram-бота на Python;
- SQLite-хранилище;
- сценарии инкубации;
- учет кормов, склад, замесы смеси, поголовье и стада;
- учет яиц и временно не несущихся кур;
- reminder runner;
- Windows-скрипты запуска и автозапуска;
- Docker, systemd-шаблоны, GitHub Actions deploy и VPS auto-deploy fallback;
- backup/restore, smoke-start, проверку диска;
- базовые тесты сервисов и обработчиков.

Ограничения текущего состояния:

- модель кормов исторически разделена на старые `feed_stocks` и актуальную складскую модель `stock_items`;
- погодный API для раздела `Яйца` подключен через Open-Meteo, но коэффициенты пока ориентировочные и не обучаются на фактической статистике хозяйства;
- нет пользовательского выбора исторической даты для записи яиц, запись идет за текущий день;
- нет полноценного web/admin-интерфейса, управление идет через Telegram и scripts;
- часть продуктовых правил требует ручной проверки на реальном хозяйстве.

## 4. Границы MVP

### Входит в MVP

- стабильный серверный запуск;
- миграции БД;
- резервное копирование и проверенное восстановление SQLite;
- надежные напоминания без дублей;
- основные пользовательские сценарии инкубации и кормов;
- основные сценарии учета яиц;
- настройки пользователя;
- редактирование и история кормов;
- админские команды для наблюдения за ботом;
- базовая аналитика событий;
- документация для установки, эксплуатации и пользователя;
- privacy policy, terms и disclaimer.

### Не входит в MVP

- платежи;
- premium-статусы;
- free-тарифы и лимиты монетизации;
- партнерские размещения;
- white-label;
- сложная web-админка, если достаточно Telegram-команд `/admin`;
- автоматическая погодная модель яйценоскости до подключения устойчивого погодного источника;
- перенос с SQLite на PostgreSQL, если не возникнет фактической нагрузки или ограничений SQLite.

## 5. Нефункциональные требования

### 5.1. Надежность

- бот должен автоматически перезапускаться после падения;
- reminder loop не должен останавливать приложение из-за одной ошибки отправки;
- повторный запуск приложения не должен приводить к повторной отправке уже отправленных уведомлений;
- все критичные операции с БД должны быть атомарными;
- после перезапуска сервера бот должен стартовать автоматически.

### 5.2. Безопасность

- токены и секреты не хранятся в Git;
- `.env` существует только на сервере и локально у разработчика;
- в репозитории есть только `.env.example`;
- доступ к `/admin` разрешен только списку Telegram ID;
- сервисные рассылки доступны только администраторам;
- резервные копии БД не публикуются в репозиторий.

### 5.3. Эксплуатация

- логи пишутся в файл и ротируются;
- есть понятная команда проверки статуса;
- есть регламент обновления через `git pull` и restart;
- есть уведомление администратору при критической ошибке или падении;
- есть мониторинг свободного места на сервере;
- есть расписание бэкапов.

### 5.4. Поддерживаемость

- изменения схемы БД оформляются миграциями;
- бизнес-контент вынесен из кода;
- пользовательские сценарии покрыты тестами хотя бы по критичным путям;
- документация обновляется вместе с изменениями поведения.

## 6. Требования к запуску и деплою

### 6.1. Серверный запуск

Необходимо перенести production-запуск с локального Windows-режима на VPS или другой стабильный сервер.

Основной рекомендуемый вариант: Docker Compose.

Допустимый вариант: systemd-сервис без Docker, если это проще для текущей инфраструктуры.

Требования:

- отдельный production-запуск без зависимости от `start-bot.ps1`;
- автоматический restart при падении;
- автозапуск после перезагрузки сервера;
- отдельные окружения `dev` и `prod`;
- разные файлы конфигурации для окружений;
- документация по первому запуску и обновлению.

### 6.2. Конфигурация

В `.env.example` должны быть описаны все переменные:

- `BOT_TOKEN`;
- `BOT_TIMEZONE`;
- `DATABASE_PATH`;
- `LOG_LEVEL`;
- `ADMIN_IDS`;
- `ENVIRONMENT`;
- `BACKUP_DIR`;
- `REMINDER_INTERVAL_SECONDS`, если интервал настраивается.

Критерии приемки:

- production-сервер запускает бота одной командой;
- после перезагрузки VPS бот поднимается без ручного вмешательства;
- токен бота отсутствует в истории Git;
- локальный dev-запуск не использует production-БД.

## 7. Требования к базе данных и миграциям

### 7.1. Миграции

Необходимо отказаться от изменения схемы через `_ensure_column` как основного механизма.

Допустимые решения:

- Alembic;
- простая собственная таблица `schema_migrations`.

Для текущего размера проекта предпочтительно начать с простой таблицы `schema_migrations`, если не планируется быстрый переход на SQLAlchemy.

Обязательные миграции:

- `001_initial` - текущая схема БД;
- последующие миграции - только через отдельные файлы;
- миграция для `notification_log`;
- миграции для пользовательских настроек;
- миграции для истории кормов и групп птицы;
- миграции для контента, если он хранится в БД.

### 7.2. Резервное копирование

Требования:

- регулярный бэкап SQLite по расписанию;
- бэкап выполняется безопасно, без повреждения активной БД;
- хранится несколько последних копий;
- есть ручная команда создания бэкапа;
- есть инструкция восстановления;
- восстановление проверено на тестовой БД.

Критерии приемки:

- новая установка может создать БД через миграции с нуля;
- существующая БД обновляется миграциями без потери данных;
- повторный запуск миграций безопасен;
- восстановленная из бэкапа БД запускается в dev-окружении.

## 8. Требования к напоминаниям

### 8.1. Типы напоминаний

Напоминания должны быть разделены по типам:

- `incubation` - инкубация;
- `feed` - корма;
- `post_hatch_care` - уход после вывода;
- `service` - сервисные сообщения, если понадобится.

### 8.2. Журнал уведомлений

Добавить таблицу `notification_log`.

Минимальные поля:

- `id`;
- `user_id`;
- `batch_id`, nullable;
- `feed_id`, nullable;
- `type`;
- `event_key`;
- `scheduled_for`;
- `sent_at`;
- `status`;
- `error_code`;
- `error_message`;
- `created_at`.

`event_key` должен однозначно описывать событие, например `incubation:batch_15:day_7:2026-05-25`.

Требования:

- одно и то же событие не отправляется повторно;
- результат каждой попытки отправки фиксируется;
- Telegram `blocked` и `deactivated` помечают пользователя неактивным;
- сетевые ошибки не выключают пользователя, но фиксируются в журнале;
- ошибки одного пользователя не блокируют отправки другим пользователям.

Критерии приемки:

- повторный запуск reminder loop в тот же день не создает дубль;
- при блокировке бота пользователь становится неактивным для рассылок;
- администратор может увидеть количество ошибок отправки;
- есть тесты на отсутствие дублей.

## 9. Пользовательские сценарии

### 9.1. Общие требования к UX

- у каждого сценария есть понятный возврат в главное меню;
- при выходе в меню очищается FSM-состояние;
- кнопки не должны быть двусмысленными;
- ошибка ввода возвращает пользователя на понятный шаг;
- пользователь не должен попадать в тупиковое состояние;
- критичные действия требуют подтверждения;
- тексты должны быть короткими, конкретными и одинаково оформленными.

### 9.2. Обязательные сценарии для ручной проверки

Перед публичным запуском необходимо вручную пройти:

- `/start`;
- добавить партию;
- изменить дату закладки;
- изменить количество яиц;
- завершить вывод;
- вернуть партию из истории;
- добавить корм;
- отредактировать корм;
- пополнить корм;
- списать корм;
- удалить корм;
- открыть календарь;
- открыть уход после вывода;
- включить и выключить напоминания;
- изменить время напоминаний;
- вернуться в меню из каждого вложенного шага.

Критерии приемки:

- каждый сценарий завершается ожидаемым состоянием в БД;
- после `/cancel` и возврата в меню нет старого FSM-состояния;
- неверный ввод не ломает сценарий;
- пользовательские сообщения не содержат технических ошибок.

## 10. Система контента

### 10.1. Вынос контента из кода

Необходимо вынести в JSON/YAML-файлы или таблицы:

- режимы инкубации по видам птицы;

- рекомендации по уходу после вывода;

Предпочтительный вариант для MVP: YAML или JSON в репозитории с версионированием.

Требования:

- изменение контента не требует переписывания бизнес-логики;
- некорректный контент должен выявляться тестом или валидатором.

### 10.2. Медицинские и ветеринарные рекомендации

Требования:

- добавить disclaimer;
- не давать жестких инструкций по лекарствам без пометки: "консультация с ветспециалистом обязательна";
- отделять общие рекомендации по уходу от медицинских назначений;
- потенциально опасные советы формулировать осторожно и справочно.

## 11. Требования к кормам

### 11.1. Функции учета кормов

Добавить:

- редактирование корма;
- название корма;
- количество птиц;
- группу птицы;
- расход на птицу;
- порог предупреждения;
- несколько кормов на одно хозяйство;
- историю пополнений;
- историю списаний;
- расчет "хватит до даты";
- предупреждение "корм закончится через N дней";
- отдельное хранение рецептов смесей.

### 11.1.1. Складовой замес смеси

Раздел `Корма -> Смесь` должен быть коротким рабочим экраном, а не списком складских остатков.

Требования:

- раздел `Корма` не должен выводить подробный список складских остатков; он показывает только короткий статус: запасы есть/нет, сколько полных замесов можно сделать, сколько кг смеси получится, что ограничивает замес;
- кнопка `Добавить корм` в разделе `Корма` должна добавлять позицию на склад, а не создавать параллельную старую карточку корма;
- команда `/feed` должна вести в тот же складской сценарий добавления позиции;
- при открытии `Смесь` бот сразу считает по текущим остаткам склада, сколько полных замесов смеси можно сделать;
- расчет учитывает выбранную или автоматически лучшую зерновую основу: `Пшеница` или `Зерносмесь`;
- автоматический выбор основы делается только по фактическому наличию на складе и максимальному числу полных замесов; цена, качество и предпочтительность ингредиента не учитываются;
- если пшеницы нет, но есть зерносмесь, бот должен выбирать зерносмесь и явно писать, что вариант выбран по текущим складским остаткам;
- если доступны оба варианта, бот выбирает тот, который дает больше полных замесов; при равенстве допустимо оставить вариант по умолчанию;
- бот показывает максимум полных замесов, примерный выход готовой смеси в кг и ограничивающий ингредиент;
- подробный список остатков остается только в разделе `Склад`;
- если полного замеса не хватает, бот показывает короткий список ингредиентов, которые нужно докупить;
- если доступно 0 полных замесов, бот не должен показывать кнопки создания замеса; допустимы только переключение основы, ручной ввод и отмена;
- для перевода ингредиентов в готовую смесь бот предлагает быстрые кнопки `Сделать 1`, `Сделать 2`, `Сделать 3` и далее до разумного лимита, а при большом запасе - кнопку `Сделать максимум`;
- кнопки создания замеса списывают ингредиенты со склада и добавляют готовую `Смесь для кур`;
- должен оставаться ручной ввод количества замесов для нестандартных случаев.
- если пользователь нажал старую кнопку замеса, а остатки уже изменились, бот должен объяснить, что ингредиентов недостаточно или остатки изменились, и предложить открыть актуальный расчет заново.

Критерии приемки:

- пользователь видит ответ на вопрос "сколько смеси я могу сделать сейчас" без перехода в подробный склад;
- при нажатии быстрой кнопки создается замес с корректным списанием ингредиентов;
- вариант с зерносмесью работает, если пшеницы нет, но зерносмесь есть на складе;
- премикс можно добавлять в граммах и пачках по 0.5 кг.
- старый callback `feeds:add` не ведет в отдельный учет кормов, а перенаправляет в складское добавление покупки.
- команда `/feed` не ведет в отдельный учет кормов, а перенаправляет в складское добавление покупки.

### 11.2. Модель данных кормов

Рекомендуемые сущности:

- `bird_groups` - группы птицы пользователя;
- `feed_stocks` - текущие остатки;
- `feed_transactions` - пополнения и списания;
- `feed_recipes` - рецепты смесей;
- `feed_recipe_items` - состав рецептов.

Критерии приемки:

- пользователь может вести несколько кормов;
- каждое изменение остатка попадает в историю;
- расчет даты окончания учитывает расход и количество птиц;
- предупреждение не дублируется каждый запуск;
- удаление корма не ломает историю, предпочтительно использовать архивирование.

## 11.3. Учет яиц

Раздел `Яйца` должен быть третьим основным разделом главного меню.

Реализованные базовые требования:

- ежедневная запись количества собранных яиц;
- история за последние дни;
- расчет за сегодня, 7 дней и 30 дней;
- прогноз на следующую неделю по среднему сбору;
- расчет только по взрослым несушкам;
- исключение из расчета наседок, кур с цыплятами, линяющих или больных кур;
- хранение города и координат;
- загрузка дневной погоды через Open-Meteo;
- ориентировочная погодная поправка к недельному прогнозу.

Следующие требования остаются актуальными:

- запись яиц за произвольную дату;
- редактирование и удаление ошибочной записи;
- привязка исключения к конкретной группе поголовья;
- автоматическое завершение исключений после `expected_until` или понятная ручная подсказка;
- ручной ввод фактической температуры в курятнике;
- настройка включения/выключения погодной поправки;
- график/агрегаты по неделям и месяцам;
- экспорт статистики яиц.

## 12. Настройки пользователя

Добавить настройки:

- часовой пояс;
- время ежедневных напоминаний;
- включение и выключение типов уведомлений;
- язык интерфейса, если появится второй язык;
- единицы измерения;
- название хозяйства;
- активность пользователя для рассылок.

Критерии приемки:

- настройки применяются к напоминаниям;
- пользователь может изменить время уведомлений без обращения к администратору;
- отключенный тип уведомлений не отправляется;
- заблокировавший бота пользователь не участвует в отправках.

## 13. Админские функции

Добавить команду `/admin`, доступную только списку ID из конфигурации.

Функции MVP:

- количество пользователей;
- количество активных пользователей;
- количество активных партий;
- ошибки отправки уведомлений;
- последние регистрации;
- последние критические ошибки, если логируются в БД;
- сервисная рассылка;
- выгрузка простой статистики.

Требования:

- неадминистратор не получает доступ к командам;
- массовая рассылка требует подтверждения;
- рассылка уважает статус неактивных пользователей;
- результат рассылки фиксируется в журнале.

## 14. Аналитика

Для MVP достаточно писать события в SQLite.

События:

- старт бота пользователем;
- создание партии;
- изменение партии;
- завершение партии;
- возврат партии из истории;
- добавление корма;
- пополнение корма;
- списание корма;
- возврат пользователя;
- ошибка сценария;
- включение или выключение уведомлений.

Рекомендуемая таблица: `analytics_events`.

Минимальные поля:

- `id`;
- `user_id`;
- `event_name`;
- `entity_type`;
- `entity_id`;
- `payload_json`;
- `created_at`.

Дальнейшее развитие после MVP:

- PostHog;
- Metabase;
- простая web-админка;
- отдельное хранилище аналитики.

## 15. Тестирование

### 15.1. Автотесты

Покрыть:

- сервисы инкубации;
- сервисы кормов;
- reminder runner;
- отсутствие дублей уведомлений;
- обработку Telegram-ошибок;
- FSM-сценарии критичных путей;
- миграции;
- smoke test старта приложения без Telegram polling;
- валидацию JSON/YAML-контента.

### 15.2. Ручное тестирование

Перед публичным запуском выполнить чек-лист из раздела 9.2 на чистой dev-БД и на БД после миграции.

Критерии приемки:

- автотесты проходят локально;
- миграции проходят на копии production-БД;
- ручной сценарий не выявляет тупиковых состояний;
- reminder loop протестирован на контролируемой дате.

## 16. Публичная документация

Подготовить:

- README для разработчика;
- инструкцию установки на сервер;
- `.env.example` со всеми переменными;
- описание команд пользователя;
- описание бэкапа;
- описание восстановления;
- changelog;
- privacy policy;
- terms of use;
- disclaimer по ветеринарным рекомендациям.

Документация должна быть достаточной, чтобы другой разработчик мог развернуть бота на новом сервере без устных пояснений.

## 17. Серверная эксплуатация

Необходимо реализовать или описать:

- ротацию логов;
- бэкапы БД по расписанию;
- мониторинг процесса;
- уведомление админу при падении;
- проверку свободного места;
- безопасное обновление через `git pull` и restart;
- rollback-процедуру на предыдущую версию;
- ручную проверку состояния после деплоя.

Минимальный checklist деплоя:

1. Проверить `git status`.
2. Забрать новую версию.
3. Установить зависимости.
4. Запустить миграции.
5. Запустить тесты или smoke test.
6. Перезапустить сервис.
7. Проверить статус сервиса.
8. Проверить последние логи.
9. Отправить тестовую команду боту.
10. Проверить, что бэкап создается.

## 18. Этапы работ

### Этап 1. Техническая стабильность

Цель: бот работает на сервере без ручного вмешательства.

Работы:

- деплой на VPS;
- Docker Compose или systemd;
- разделение dev/prod;
- миграции;
- бэкапы;
- восстановление из бэкапа;
- стабильный reminder loop;
- единый production-запуск;
- базовая ротация логов.

Критерий завершения:

- бот переживает перезагрузку сервера;
- миграции применяются с нуля и поверх существующей БД;
- есть проверенный бэкап и восстановление;
- напоминания не дублируются.

### Этап 2. Стабильный пользовательский MVP

Цель: обычный пользователь проходит все ключевые сценарии без помощи разработчика.

Работы:

- полная проверка сценариев;
- настройки пользователя;
- редактирование кормов;
- история кормов;
- группы птицы;
- понятные возвраты в меню;
- очистка FSM-состояний;
- улучшение текстов и кнопок;
- вынос контента из кода;
- disclaimer.

Критерий завершения:

- все сценарии из чек-листа пройдены;
- пользователь может настроить уведомления;
- корма можно редактировать и смотреть историю;
- контент меняется без изменения бизнес-логики.

### Этап 3. Публичность

Цель: бот готов к первым внешним пользователям.

Работы:

- `/admin`;
- аналитика;
- документация;
- privacy policy;
- terms/disclaimer;
- мониторинг;
- уведомления админу о критических сбоях;
- changelog.

Критерий завершения:

- администратор видит базовую статистику;
- есть публичные документы;
- ошибки отправки и регистрации пользователей наблюдаемы;
- эксплуатация описана документом.

### Этап 4. Подготовка к монетизации

Цель: подготовить основу для коммерческих функций после стабильного MVP.

Работы:

- ограничения free-тарифа;
- учет premium-статуса;
- платежи;
- партнерские размещения;
- white-label.

Важно: монетизацию не начинать до завершения этапов 1-3. До этого приоритет - стабильность, доверие пользователей и отсутствие ручного сопровождения.

## 19. Рекомендуемый порядок реализации

1. Ввести миграции и зафиксировать текущую схему как `001_initial`.
2. Добавить `notification_log` и защиту от дублей.
3. Разделить dev/prod-конфигурацию.
4. Подготовить Docker Compose или systemd.
5. Настроить бэкапы и восстановление.
6. Пройти ручной чек-лист пользовательских сценариев.
7. Доработать корма и настройки пользователя.
8. Вынести контент из кода.
9. Добавить `/admin` и аналитику.
10. Оформить публичную документацию.

## 20. Критерии готовности публичного MVP

MVP считается готовым, когда:

- бот развернут на стабильном сервере;
- production-запуск не зависит от ручных Windows-скриптов;
- токены не находятся в Git;
- миграции работают;
- бэкапы создаются и восстановление проверено;
- напоминания не дублируются;
- Telegram-ошибки обрабатываются;
- пользователь может пройти все основные сценарии;
- есть настройки уведомлений;
- корма можно редактировать и отслеживать в истории;
- есть админский доступ к базовой статистике;
- есть публичная документация и disclaimer;
- автотесты и smoke test проходят;
- есть понятная инструкция эксплуатации.
