# ТЗ: web-версия tg_bot_inkubator

## Цель

Добавить web-интерфейс к текущему Telegram-боту инкубатора, кормов, склада,
стад и яиц. Web-версия должна использовать существующую бизнес-логику проекта,
не ломать Telegram-бота и не запускаться внутри polling-процесса.

Первый релиз web-версии должен быть безопасным: отдельный сервис, авторизация,
статус, сводка и read-only просмотр основных данных. Управляющие действия
добавляются после того, как read-only кабинет стабилен.

## Архитектурное решение

Web запускается отдельным процессом рядом с Telegram-ботом:

```text
Telegram bot -> services/repositories -> SQLite
Web app      -> services/repositories -> SQLite
Status probe -> SQLite read-only
```

Правила:

- не встраивать web в `app/main.py`;
- не добавлять web endpoints в Telegram polling-процесс;
- если web упал, Telegram-бот должен продолжать работать;
- расчеты кормов, стад, яиц и инкубации не дублировать в web-слое, а брать из
  существующих `services`;
- первый этап делать преимущественно read-only;
- write endpoints добавлять только после авторизации и тестов.

## Технологический вариант

Рекомендуемый backend:

- FastAPI;
- отдельный entrypoint: `scripts/web_app.py` или `app/web/main.py`;
- SQLite через существующие repositories;
- шаблоны или небольшой frontend поверх API.

Рекомендуемая конфигурация:

```env
WEB_ENABLED=false
WEB_HOST=127.0.0.1
WEB_PORT=8080
WEB_ADMIN_TOKEN=
```

По умолчанию web должен слушать только `127.0.0.1`. Наружу открывать только
через reverse proxy, TLS и авторизацию.

## Этап 1. Web foundation

Задача: поднять отдельный web-сервис и минимальный безопасный интерфейс.

Состав работ:

- добавить web entrypoint;
- добавить FastAPI-приложение;
- добавить token-авторизацию;
- добавить health/status/version endpoints;
- добавить минимальную главную страницу;
- подключить существующий `status_probe` для статуса;
- описать запуск в документации;
- покрыть базовыми тестами.

Минимальные endpoints:

```text
GET /health
GET /status
GET /version
GET /
```

`GET /health`:

```json
{
  "status": "ok"
}
```

`GET /status`:

- использует `app.services.status_probe`;
- показывает `polling_bot`;
- показывает `reminder_runner`;
- показывает состояние БД;
- показывает критичные ошибки;
- не пишет в БД.

`GET /version`:

- версия;
- commit;
- последний запуск;
- последний деплой;
- GitHub/changelog.

`GET /`:

- простая web-сводка;
- статус сервисов;
- версия;
- ссылки на основные разделы.

Критерии приемки:

- web запускается отдельной командой;
- Telegram-бот продолжает запускаться отдельно;
- `/health` доступен;
- закрытые страницы без токена недоступны;
- закрытые страницы с токеном доступны;
- `/status` не меняет SQLite;
- тесты проходят.

## Этап 2. Read-only кабинет

Задача: показать в web текущий объем данных без возможности менять их.

Разделы:

### Сводка

- статус сервисов;
- версия;
- последний запуск;
- последний деплой;
- погода;
- яйца за сегодня;
- активные партии инкубации;
- запас готовой смеси;
- ближайшие закупки.

### Корма

- склад ингредиентов;
- готовая смесь;
- сколько замесов можно сделать;
- сколько кг смеси получится;
- ограничивающий ингредиент;
- история операций.

### Смесь

- рецепт в частях;
- базовый замес;
- потенциальное количество замесов;
- текущий запас готовой смеси;
- чеклист только для просмотра на первом read-only этапе.

### Поголовье

- группы птиц;
- тип: несушки, петухи, цыплята, смешанная группа;
- количество;
- даты вывода/подсадки;
- активность/архив.

### Стада

- список стад;
- состав стада;
- назначенная смесь;
- расчет расхода в день;
- на сколько дней хватит готовой смеси;
- на сколько дней хватит ингредиентов для новых замесов.

### Яйца

- сборы по дням;
- сегодня/вчера;
- прогноз на 7 дней;
- исключенные несушки;
- погода день/ночь и кратко завтра.

### Инкубация

- активные партии;
- даты закладки;
- ожидаемые даты вывода;
- этапы;
- напоминания.

### Настройки

- хозяйство;
- город погоды;
- разделы Telegram-бота;
- уведомления;
- о боте;
- технический статус для администратора.

Критерии приемки:

- все разделы открываются без записи в БД;
- пустые данные отображаются нормально;
- расчеты совпадают с Telegram-ботом;
- названия русифицированы;
- технические поля не торчат пользователю без необходимости;
- тесты проходят.

## Этап 3. Управляющие действия

Добавлять только после стабильного read-only кабинета.

Планируемые операции:

```text
POST /eggs/entries
POST /stock/purchases
POST /feeds/mixes
POST /bird-groups
PATCH /bird-groups/{id}
POST /flocks
PATCH /flocks/{id}
POST /settings/weather
PATCH /settings/sections
```

Правила:

- все write endpoints используют текущие services/repositories;
- SQL в обработчиках web-слоя не писать;
- опасные действия требуют подтверждения;
- ошибки валидации должны быть понятны пользователю;
- Telegram-бот и web должны видеть одни и те же данные.

Критерии приемки:

- можно добавить яйца за сегодня/вчера;
- можно добавить покупку на склад;
- можно создать замес;
- можно создать/изменить поголовье;
- можно создать/изменить стадо;
- можно изменить город погоды;
- параллельная работа Telegram и web не ломает данные.

## Этап 4. Полноценный web UX

Задача: превратить read-only кабинет и базовые формы в удобную рабочую панель.

Требования:

- интерфейс не должен копировать Telegram-кнопки;
- web должен быть обзорным, табличным и быстрым;
- использовать русские названия без технических англицизмов;
- формы должны иметь подтверждения и понятные ошибки;
- таблицы должны поддерживать фильтры и сортировку там, где это нужно;
- мобильная версия должна быть пригодна для быстрого просмотра и простых
  действий;
- админский технический статус отделить от хозяйственных разделов.

## Авторизация и безопасность

Минимум для первого этапа:

- `WEB_ADMIN_TOKEN`;
- закрыть все хозяйственные и статусные страницы;
- `/health` можно оставить публичным только если он не раскрывает данные.

Позже:

- вход по Telegram ID из `ADMIN_IDS`;
- роли пользователей;
- аудит write-действий.

Запрещено:

- показывать `BOT_TOKEN`;
- показывать секреты окружения;
- отдавать персональные данные без авторизации;
- открывать web наружу без TLS/reverse proxy;
- добавлять write endpoints без тестов.

## Тесты

Этап 1:

- `/health` возвращает `ok`;
- `/status` возвращает валидный JSON;
- `/status` не пишет в БД;
- закрытые страницы без токена недоступны;
- закрытые страницы с токеном доступны.

Этап 2:

- сводка собирается без падения;
- корма читаются корректно;
- яйца читаются корректно;
- стада читаются корректно;
- инкубация читается корректно;
- пустая БД отображается нормально.

Этап 3:

- добавление яиц;
- добавление покупки;
- создание замеса;
- создание поголовья;
- создание стада;
- изменение настроек;
- ошибки валидации.

## Узкие места

1. Дублирование Telegram UX.
   Web не должен быть чатом в браузере. Нужна рабочая панель.

2. SQLite и конкурентные записи.
   На read-only этапе риск низкий. На write-этапе нужно тестировать совместную
   работу Telegram-бота и web.

3. Дублирование бизнес-логики.
   Расчеты нельзя переписывать отдельно для web. Нужно использовать общий
   service-слой.

4. Авторизация.
   Web становится новой точкой доступа к данным хозяйства, поэтому защита нужна
   с первого этапа.

5. Объем.
   Полную web-версию нельзя делать одним большим коммитом. Работу нужно вести
   этапами.

## Рабочий порядок разработки

Каждый этап выполняется отдельным циклом:

1. Сформулировать подзадачи текущего этапа.
2. Реализовать подзадачи.
3. Запустить тесты и ручную проверку, если нужен локальный web.
4. Провести саморевью diff.
5. Написать отчет:
   - что сделано;
   - что проверено;
   - что изменилось в UX/архитектуре;
   - какие узкие места остались;
   - какой следующий этап предлагается.
6. Остановиться и ждать команды на следующий этап.

После выполнения всех этапов:

- составить схему выполненной работы;
- описать итоговую архитектуру;
- перечислить оставшиеся риски;
- предложить дальнейшие доработки.

## Рекомендуемый первый цикл

Начать с этапа 1:

```text
1. Добавить FastAPI-зависимость, если ее нет.
2. Создать app/web/main.py.
3. Создать scripts/web_app.py.
4. Подключить конфиг WEB_HOST/WEB_PORT/WEB_ADMIN_TOKEN.
5. Сделать /health, /status, /version, /.
6. Сделать простую HTML-сводку.
7. Добавить тесты.
8. Обновить документацию запуска.
9. Провести саморевью и остановиться с отчетом.
```
