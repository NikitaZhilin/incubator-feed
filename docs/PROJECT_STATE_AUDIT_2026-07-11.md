## Функциональная карта

### Инкубация

Реализованы создание партий, активные партии, календарь, план на сегодня, история, статистика вывода, режимы по видам птицы, уход после вывода, завершение и возврат партии в активные. Контент вынесен в JSON. Web-слой умеет создавать партию, завершать вывод и возвращать завершенную партию.

### Корма, склад и замесы

В проекте сосуществуют две модели:

- старая модель `feed_stocks` / `feed_transactions`;
- актуальная складская модель `stock_items` / `stock_transactions` / `mix_productions` / `mix_production_items`.

Пользовательский путь сейчас ведется через складскую модель. `Корма -> Смесь` рассчитывает возможные замесы по текущим остаткам, выбирает пшеницу или зерносмесь по доступности, показывает ограничивающий ингредиент и позволяет выбрать количество замесов.

После выбора количества реализован обязательный выбор режима:

- `Сделать сейчас` - открывает чек-лист ингредиентов, после завершения списывает ингредиенты и добавляет готовую смесь на склад.
- `Записать как уже скормленный` - чек-лист не открывается; бот спрашивает дату (`Сегодня`, `7 дней назад`, `Без даты / не помню`, `Ввести дату`), создает исторический замес, списывает ингредиенты и сразу списывает полученную готовую смесь как съеденную.

Миграция `015_stock_mix_already_fed.py` добавила `mix_productions.mode`, `mix_productions.produced_at` и `stock_transactions.occurred_at`. `StockService.last_mix_output()` исключает замесы `already_fed`, чтобы они не считались текущим запасом готовой смеси.

### Поголовье и стада

Есть отдельные группы поголовья: несушки, петухи, смешанные взрослые группы и цыплята. Для цыплят учитываются дата вывода, дата подсадки и возрастные нормы расхода. Стада собираются из групп и получают назначение готовой смеси. Расход считается по составу стада, доле рациона и запасу.

### Яйца

Раздел `Яйца` включает обычный сбор, сбор за несколько дней, историю, расчеты, временно не несущихся кур, город и погоду.

Фактическая логика:

- `Обычный сбор` в Telegram записывает `Сегодня` или `Вчера`.
- `Сбор за несколько дней` принимает дату фактического сбора, затем общее количество, затем период вручную или `Не помню период`.
- Ручной период распределяет любое количество яиц на `2..30` дней назад от даты фактического сбора через целое деление, остаток получает самая свежая дата.
- Неизвестный период считает среднее по дням с записями, берет `ceil(total / average)`, ограничивает `2..30` и раскладывает яйца по ближайшим пустым дням назад от даты фактического сбора.
- Перед сохранением показывается предпросмотр.
- В БД создаются обычные строки `egg_entries`; отдельная таблица для много-дневных сборов не нужна.
- Telegram умеет исправлять количество и дату записи через историю. Удаление записи реализовано в web.
- Web умеет добавить запись за выбранную дату через `entry_date`, а также редактировать и удалять запись.

Статистика использует `SUM(eggs_count)` и `GROUP BY entry_date`, поэтому несколько записей на одну дату корректно агрегируются.

### Погода и яйценоскость

Погода подключена через Open-Meteo без API key. Есть fallback на `wttr.in` по координатам. Сохраняются дневные, ночные и завтрашние значения. Погодная поправка к прогнозу яйценоскости намеренно простая и показывается как ориентировочная.

### Птицевод

Раздел `Птицевод` реализован как детерминированный советник по данным хозяйства. Он показывает план на сегодня, блок по кормам и замесу, срочность следующего замеса, анализ просадки яиц, задачи по инкубации и безопасный сценарий при проблемах с птицей. Советник не ставит диагнозы и не назначает лекарства.

### Напоминания и служебные сообщения

`ReminderRunner` отправляет ежедневные напоминания и сводку, учитывает пользовательский часовой пояс и настройки разделов. Дубли блокируются через `notification_log`. Ошибки Telegram обрабатываются: заблокированные/деактивированные пользователи помечаются неактивными. Release notice и admin startup notice разведены по назначению и дедуплицируются.

### Web-кабинет и Mini App база

FastAPI web-процесс поддерживает:

- `/`, `/summary`, `/status`, `/version`;
- `/feeds`, `/mix`, `/livestock`, `/eggs`, `/incubation`, `/about`;
- JSON endpoints для основных страниц;
- write-сценарии: покупка на склад, корректировка остатка, подтвержденный замес, создание/редактирование поголовья, создание/редактирование стада, назначение смеси, добавление/редактирование/удаление яиц, изменение города погоды, включение/выключение разделов, создание/завершение/возврат инкубационной партии.

Ограничение: много-дневный сбор яиц и режим `уже скормленный` для замеса реализованы в Telegram. Для web parity им нужны отдельные формы/endpoint с предпросмотром и подтверждением.

## Актуальные разрывы и риски

- Нужно консолидировать старую модель `feed_stocks` и актуальную складскую модель `stock_items`, чтобы пользовательские пути и документация не расходились.
- Telegram и web parity не полные: web пока не имеет много-дневного сбора яиц и режима замеса `уже скормленный`; Telegram не имеет удаления записи яиц, хотя web имеет.
- В Telegram обычный сбор яиц напрямую предлагает только `Сегодня`/`Вчера`; произвольная дата доступна через web или через исправление записи.
- Исключения несушек пока не привязаны к конкретной группе поголовья.
- Погодная поправка не обучается на фактической истории хозяйства и должна оставаться ориентировочной.
- Рецепт смеси фактически живет в JSON-контенте, а таблицы рецептов в БД остаются заделом.
- Web/Mini App hardening, сессии и production-модель авторизации требуют отдельного доведения по `docs/FINAL_HARDENING_BACKLOG.md`.
- Документацию нужно обновлять вместе с каждым релизом: этот аудит исправил устаревшие места после последних доработок.
