# Roadmap and Current Gaps

Документ фиксирует актуальные узкие места после сверки документации с текущей реализацией. Это не список обязательств на один релиз, а рабочая карта следующих доработок.

## 1. Консолидация кормов и склада

Сейчас в проекте сосуществуют две модели:

- старая карточка корма `feed_stocks`;
- актуальная складская модель `stock_items` с покупками, замесами и назначениями.

Что стоит сделать:

- определить один основной пользовательский путь и постепенно спрятать старые карточки корма;
- оставить миграционную совместимость, но убрать дублирующие UX-сценарии;
- в документации и интерфейсе закрепить, что `Добавить корм` означает складскую покупку;
- добавить тест, что все публичные кнопки добавления корма ведут в склад.

## 2. Раздел `Яйца`

Базовый учет уже есть, но для практического использования нужны:

- запись яиц за выбранную дату, а не только за сегодня;
- редактирование и удаление ошибочной записи;
- привязка временно не несущейся курицы к группе поголовья;
- автоматическая подсказка, что срок исключения истек;
- экспорт статистики яиц;
- недельные и месячные агрегаты в более удобном виде;
- ручной ввод заметки к сбору яиц.

## 3. Погода и яйценоскость

Сейчас бот хранит город, но не загружает погодные данные.

Что стоит сделать:

- выбрать погодный источник с понятными лимитами и лицензией;
- добавить geocoding города;
- сохранять дневную температуру и, если доступно, влажность/осадки;
- не делать прогноз "магическим": сначала показывать погоду рядом со статистикой, затем добавлять коэффициенты только после проверки на фактических данных;
- дать пользователю возможность выключить погодное влияние.

## 4. Поголовье и стада

Сейчас `Яйца` считает только группы с ролью `hens`.

Риски:

- если пользователь создал взрослых кур как `mixed`, яйца покажут 0 несушек;
- разделение "поголовье" и "стада" уже улучшено, но нужны более понятные подсказки при создании групп.

Что стоит сделать:

- добавить предупреждение в поголовье, если есть adult/mixed группы и нет несушек;
- разрешить быстро сменить роль группы;
- в карточке стада явно показывать, какие группы входят и сколько из них несушек/петухов/цыплят;
- продумать перевод цыплят во взрослую группу после даты подсадки.

## 5. Рецепты смеси

Рецепт смеси сейчас лежит в JSON-контенте, таблицы рецептов в БД зарезервированы.

Что стоит сделать:

- решить, остаются ли рецепты в JSON или переезжают в БД;
- добавить версионирование рецепта в пользовательский экран;
- добавить пользовательское редактирование рецепта только после стабилизации текущего базового рецепта;
- хранить историю, по какой версии рецепта был создан конкретный замес.

## 6. Release notes и версия

Обычные деплои не шумят, `О боте` показывает версию и notes.

Узкое место:

- GitHub Actions при обычном push обновляет `RELEASE_NOTES` дефолтным текстом workflow. Это удобно для текущего релиза, но требует дисциплины.

Что стоит сделать:

- хранить краткие notes текущего релиза в `docs/CHANGELOG.md` или отдельном файле и подтягивать их в deploy;
- добавить в GitHub Actions явный workflow input `release_importance`, потому что сейчас workflow обновляет версию и notes, но не дает выбрать `medium/major/critical`;
- сделать `skip_release_notice=true` по умолчанию для ручных workflow runs, если не выбран уровень важности;
- разделить "версия кода" и "публичная версия продукта".

## 7. Документация и ручная приемка

Документация обновлена, но нужен регулярный процесс:

- после каждого релиза обновлять `docs/CHANGELOG.md`;
- держать README как краткую входную точку, а детали в `docs/*`;
- перед релизом проходить checklist из `docs/OPERATIONS.md`;
- добавить отдельный сценарий ручной приемки для кормов, стад и яиц.

## 8. Тесты

Сейчас покрыты сервисы, миграции и часть keyboard/helper логики.

Что стоит усилить:

- FSM-сценарии раздела `Яйца`;
- проверки, что старые callback-пути не ведут в устаревший UX;
- миграция production-like БД с существующими складскими позициями и стадами;
- edge cases по ingredient matching: разные написания кукурузы, комбикорма, премикса, зерносмеси;
- тесты на истечение срока исключения несушки.

## 9. Админка и наблюдаемость

`/admin` уже есть, но проекту станет полезно:

- видеть текущий deployed commit и время деплоя в админском экране;
- смотреть последние ошибки сценариев по пользователям;
- видеть количество активных складов/стад/записей яиц;
- иметь команду проверки миграций и состояния БД из админки.
