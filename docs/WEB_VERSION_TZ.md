# ТЗ: сайт и Telegram Mini App tg_bot_inkubator

## 1. Назначение

Документ пересобирает требования к web-версии и Telegram Mini App под текущее
состояние проекта `tg_bot_inkubator`.

Цель: дать пользователю единый интерфейс для работы с хозяйством из трех
поверхностей:

- Telegram-бот;
- обычный web-сайт;
- Telegram Mini App.

Сайт и Mini App должны иметь похожий интерфейс, общую навигационную модель и
одни и те же доменные сценарии. Они могут запускаться параллельно с Telegram
ботом, но все читают и меняют одну базу данных. Любая правка из бота, сайта или
Mini App должна быть видна во всех остальных поверхностях после обновления
данных.

## 2. Важная адаптация исходного ТЗ

Исходное ТЗ было написано для другой продуктовой области: списки, напоминания,
лекарства, водительский раздел, `/me/...`, `/app/info`, PostgreSQL и
frontend-файлы `src/web/index.html`, `src/web/app.js`, `src/web/styles.css`.

В текущем проекте это неприменимо как прямое требование. Актуальная область
проекта:

- инкубационные партии;
- календарь, план на сегодня и напоминания по инкубации;
- корма, склад, покупки и замесы;
- поголовье и стада;
- учет яиц, временно не несущиеся куры и погода;
- настройки хозяйства и уведомлений;
- админские команды и наблюдаемость.

Актуальная реализация:

- Telegram-бот: `app/main.py`, `app/handlers/*`, `app/keyboards/*`;
- web backend: `app/web/main.py`;
- web config: `app/web/config.py`;
- web entrypoint: `scripts/web_app.py`;
- доменная логика: `app/services/*`;
- хранилище: SQLite через `app/storage/repositories/*`;
- миграции: `app/storage/migrations/*`;
- тесты web-слоя: `tests/test_web_app.py`.

Поэтому новое ТЗ не требует переносить проект в `src/web`, React/Next.js или
PostgreSQL для MVP. Развитие должно идти поверх текущего FastAPI web-сервиса,
общего service/repository слоя и существующей SQLite БД.

## 3. Текущее состояние

Уже реализовано:

- отдельный FastAPI web-сервис, запускаемый через `python -B scripts/web_app.py`;
- публичный `/health`;
- защищенные `/status`, `/version`, `/summary`;
- HTML-страницы и JSON endpoints:
  - `/`;
  - `/feeds`, `/feeds/data`;
  - `/mix`, `/mix/data`, `/mix/confirm`;
  - `/livestock`, `/livestock/data`;
  - `/eggs`, `/eggs/data`;
  - `/incubation`, `/incubation/data`;
  - `/about`, `/about/data`;
- write-сценарии:
  - `POST /stock/purchases`;
  - `POST /feeds/mixes`;
  - `POST /eggs/entries`;
  - `POST /bird-groups`;
  - `PATCH|POST /bird-groups/{group_id}`;
  - `POST /flocks`;
  - `PATCH|POST /flocks/{flock_id}`;
  - `POST /flock-feed-assignments`;
  - `POST /settings/weather`;
  - `PATCH|POST /settings/sections`;
- авторизация web-страниц через `WEB_ADMIN_TOKEN` и/или `WEB_LINK_TOKEN`;
- отдельные кнопки `🌐 Открыть сайт` и `📱 Открыть Mini App` в Telegram при
  настроенном `WEB_PUBLIC_URL`;
- Telegram menu button на Mini App при HTTPS `WEB_PUBLIC_URL`;
- Docker Compose сервисы `bot`, `web`, `backup`;
- тесты на основные web endpoints, авторизацию, навигацию и базовые write-сценарии.

Не реализовано и является следующим этапом после функциональной проверки:

- Telegram Mini App runtime;
- подключение Telegram JS SDK;
- серверная валидация `initData`;
- авторизация Mini App без ручного web-ключа;
- отдельный пользовательский API в стиле `/me/...`;
- `WEB_TEST_LOGIN_ENABLED`;
- production-check команда для Mini App;
- CORS/API docs настройки как отдельные production guards.

Функциональные правила записей и расчетов вынесены в
`docs/WEB_FUNCTIONAL_RULES.md`. Задачи по авторизации, публичному запуску,
персональным web-ключам и production hardening отложены в
`docs/FINAL_HARDENING_BACKLOG.md`.

Инфраструктурная правка для текущего тестового режима уже внесена:
web-контейнер в `docker-compose.yml` монтирует `./data:/app/data` с правом
записи, потому что web уже имеет write endpoints. Если позже потребуется
read-only deployment, write endpoints нужно будет явно отключать.

## 4. Целевая архитектура

```text
Telegram Bot       -> services -> repositories -> SQLite
Web Site           -> services -> repositories -> SQLite
Telegram Mini App  -> services -> repositories -> SQLite
Status/Operations  -> repositories/status probe -> SQLite
```

Правила:

- бот, сайт и Mini App запускаются отдельными процессами/контейнерами;
- web/Mini App не встраиваются в Telegram polling process;
- при падении web-сервиса Telegram-бот должен продолжать работать;
- бизнес-правила не дублируются во frontend или обработчиках FastAPI;
- все операции записи идут через существующие `services`;
- repositories остаются единственным слоем прямого доступа к БД;
- пользовательские данные изолируются по Telegram user ID;
- админские функции не смешиваются с пользовательским Mini App.

Для MVP база остается SQLite. Переход на PostgreSQL фиксируется отдельным ТЗ,
если появится реальная нагрузка, ограничения SQLite или необходимость
многопроцессной записи с более строгими гарантиями.

## 5. Режимы запуска

### Telegram-бот

Запуск:

```powershell
python main.py
```

Основные обязанности:

- пользовательские сценарии Telegram;
- уведомления и reminder runner;
- выдача fallback-ссылки на сайт, если Mini App недоступна;
- кнопка выбора web/Mini App;
- админские команды.

### Web-сайт

Запуск:

```powershell
python -B scripts/web_app.py
```

Назначение:

- полноэкранный web-кабинет в браузере;
- тот же набор доменных разделов, что в Mini App;
- вход через защищенную ссылку `WEB_LINK_TOKEN` или другой безопасный web-login
  механизм;
- удобный desktop/tablet режим поверх той же структуры данных.

### Telegram Mini App

Запускается через HTTPS URL из Telegram. Для MVP Mini App может использовать тот
же FastAPI web-сервис и тот же HTML/CSS, но должна уметь определить Telegram
runtime, применить Telegram theme params и авторизоваться через `initData`.

## 6. Кнопка выбора в Telegram

В Telegram должна быть доступна кнопка, после нажатия на которую пользователь
видит выбор:

- `Открыть сайт`;
- `Открыть Mini App`.

Поведение:

- если `WEB_PUBLIC_URL` не настроен, показывать понятное сообщение без
  технических деталей для обычного пользователя;
- если `WEB_PUBLIC_URL` настроен как HTTPS и Mini App авторизация готова,
  `Открыть Mini App` использует `web_app=WebAppInfo(url=...)`;
- если URL не HTTPS или Mini App авторизация не включена, Mini App-кнопку не
  показывать или показывать как недоступную с коротким объяснением;
- `Открыть сайт` открывает обычный URL через `url=...`;
- если задан `WEB_LINK_TOKEN`, ссылка сайта может содержать `?auth=<token>`;
- `WEB_ADMIN_TOKEN` не должен попадать в пользовательские Telegram-ссылки.

Для личных чатов можно дополнительно настроить Telegram menu button через Bot
API `setChatMenuButton`, но это не блокер MVP.

## 7. Авторизация

На текущем тестовом этапе функционал проверяет администратор. Поэтому достаточно
текущего защищенного входа через `WEB_ADMIN_TOKEN` или закрытую ссылку с
`WEB_LINK_TOKEN`, если ссылка не передается внешним пользователям.

Полноценная пользовательская авторизация, персональные web-ключи,
`initData`-валидация Mini App и production guards отложены до финальной
отладки. Список отложенных требований: `docs/FINAL_HARDENING_BACKLOG.md`.

### Текущий web-вход

Сохраняется:

- `Authorization: Bearer <WEB_ADMIN_TOKEN>`;
- `X-Web-Token: <WEB_ADMIN_TOKEN>`;
- `?auth=<WEB_LINK_TOKEN>` для ссылочного пользовательского входа.

Требования:

- `WEB_ADMIN_TOKEN` использовать только для админского/операционного доступа;
- `WEB_LINK_TOKEN` считать временным механизмом до персональных web-ключей;
- не логировать значения токенов;
- ошибки авторизации не должны раскрывать, существует ли пользователь.

### Mini App вход

Добавить серверную валидацию Telegram `initData` перед публичным запуском Mini
App. Для текущего тестового функционального этапа это не блокер.

Требования:

- клиент передает `X-Telegram-Init-Data`;
- backend проверяет подпись `initData` по `BOT_TOKEN`;
- backend проверяет возраст auth data;
- TTL задается переменной `USER_AUTH_MAX_AGE_SECONDS`;
- при валидном `initData` backend создает или обновляет пользователя в таблице
  `users`;
- при невалидном или просроченном `initData` пользовательские endpoints
  возвращают `401`;
- raw `initData` нельзя писать в логи;
- для Mini App нельзя создавать отдельный admin-token based путь.

### Test login

Если будет добавлен тестовый вход через admin token, он должен работать только
при `WEB_TEST_LOGIN_ENABLED=true`. В production значение должно быть `false`.

## 8. Frontend и UX

Для MVP не вводить React/Next.js без отдельного решения. Сначала развивать
текущий server-rendered HTML/CSS в `app/web/main.py`, а при росте сложности
вынести static assets отдельным этапом.

Общие требования:

- mobile-first;
- сайт и Mini App используют одинаковую терминологию, порядок разделов и
  похожие controls;
- на desktop сайт может быть более табличным и обзорным;
- в Mini App приоритет у компактного mobile flow;
- не использовать browser-native `alert`, `prompt`, `confirm`;
- ошибки показывать inline рядом с действием или в общем message-блоке;
- критичные удаления требуют второго подтверждающего клика;
- длинные названия партий, кормов, поголовья, стад и заметок не должны ломать
  layout;
- не допускать горизонтального скролла страницы на мобильных экранах, кроме
  осознанных scroll-контейнеров таблиц;
- все тексты MVP на русском.

Telegram Mini App требования:

- подключить Telegram JS SDK в `<head>` до локального клиентского JS, если
  frontend будет вынесен в static assets;
- вызывать `Telegram.WebApp.ready()` после подготовки UI;
- использовать `Telegram.WebApp.expand()`, где это улучшает основной mobile
  flow;
- учитывать `viewportStableHeight`, safe area и Telegram theme params;
- синхронизировать цвета с Telegram theme params;
- иметь fallback светлой/темной темы вне Telegram;
- при валидном `initData` скрывать ручной login/fallback panel;
- использовать Telegram BackButton там, где он заменяет внутреннюю кнопку
  `Назад`;
- показывать launch/auth ошибки inline.

## 9. Доменные разделы MVP

### Сводка

Показывать:

- статус хозяйства;
- яйца за сегодня и последние дни;
- активные партии инкубации;
- ближайшие действия и напоминания;
- остатки готовой смеси;
- ближайшие риски по кормам;
- версию, канал релиза, changelog;
- тестовое предупреждение, если оно включено.

Обычный пользователь не должен видеть админскую сводку.

### Инкубация

Требования:

- просмотр активных и завершенных партий;
- создание партии;
- просмотр дня инкубации, этапа, режима, ожидаемой даты вывода;
- план на сегодня;
- календарь работ;
- завершение вывода;
- возврат завершенной партии в активные;
- рекомендации только как учетные подсказки, без ветеринарных обещаний.

### Корма и склад

Требования:

- просмотр складских позиций;
- создание покупки;
- история операций;
- фильтрация и сортировка;
- расчет дневного расхода;
- расчет, на сколько дней хватит остатков;
- отображение ограничивающего ингредиента для замеса;
- понятная работа с единицами: кг, граммы, мешки, пачки.

### Смесь

Требования:

- просмотр рецепта;
- выбор основы зерна, если доступно несколько вариантов;
- расчет доступного числа замесов;
- экран подтверждения перед списанием ингредиентов;
- создание замеса через `StockService`;
- история замесов;
- в будущем: версионирование рецептов.

### Поголовье и стада

Требования:

- просмотр групп птиц;
- создание и редактирование группы;
- роли: несушки, петухи, смешанная взрослая группа, цыплята;
- даты вывода и подсадки;
- резерв расхода;
- создание и редактирование стада;
- выбор групп, входящих в стадо;
- назначение смеси стаду;
- сводка по расходу стада и срокам остатков.

### Яйца

Требования:

- просмотр сбора за сегодня, вчера и историю;
- добавление записи сбора;
- запись за выбранную дату добавить отдельным следующим шагом;
- редактирование и удаление ошибочной записи добавить отдельным следующим шагом;
- учет временно не несущихся кур;
- подсказка по истекшему сроку исключения;
- сохраненная погода и город;
- изменение города погоды;
- погодные формулировки показывать как ориентировочные.

### Настройки

Показывать и редактировать:

- название хозяйства, если сценарий уже доступен в Telegram;
- часовой пояс;
- время уведомлений;
- включение/выключение разделов;
- город погоды;
- версия, канал релиза, changelog, GitHub;
- Telegram ID текущего пользователя.

Выдачу web-ключей оставить в Telegram-боте или админском защищенном сценарии.
Публичную кнопку `Создать ключ` внутри Mini App не делать без отдельной защиты.

### Админская зона

Админские функции остаются в Telegram `/admin` и/или отдельном защищенном web
разделе. Они не должны быть частью основной пользовательской навигации Mini App.

## 10. Backend требования

Сохранить текущий слой:

```text
FastAPI routes -> services -> repositories -> SQLite
Telegram handlers -> services -> repositories -> SQLite
```

Требования:

- все операции записи повторно проверяют user scope на backend;
- нельзя доверять user_id из формы без проверки доступа;
- для Mini App текущий пользователь определяется через валидированный
  `initData`;
- для web-ссылки текущий пользователь должен быть связан с персональным ключом,
  когда появятся персональные web-ключи;
- SQL не писать в FastAPI handlers, если уже есть repository/service;
- ошибки service-слоя переводить в понятные пользовательские сообщения;
- `/health` остается публичным и не раскрывает данные;
- `/status` и runtime details доступны только защищенному доступу.

Если вводится JSON API для Mini App, придерживаться same-origin endpoints:

- `GET /app/info`;
- `GET /me/summary`;
- `GET /me/feeds`;
- `GET /me/mix`;
- `GET /me/livestock`;
- `GET /me/eggs`;
- `GET /me/incubation`;
- `GET /me/settings`;
- write endpoints под `/me/...` только после серверной проверки владельца.

Admin API не вызывать из пользовательского Mini App.

## 11. Deployment

Production требования:

- `WEB_PUBLIC_URL` должен быть HTTPS URL без trailing slash;
- Telegram `web_app` кнопки использовать только с HTTPS URL;
- web и bot запускаются отдельными контейнерами/процессами;
- запускать только один bot polling instance;
- для write-сценариев web-контейнер должен иметь write-доступ к БД;
- если web-контейнер оставлен read-only, все write endpoints должны быть
  отключены или недоступны;
- web должен быть закрыт reverse proxy/firewall;
- SQLite файл и backup директории не должны быть публично доступны;
- `.env`, `.env.prod`, БД, логи и бэкапы не попадают в Git;
- CORS в production не должен быть `*`, если будет добавлен cross-origin API;
- API docs в production отключить, если будут включены FastAPI docs наружу.

Обновить deployment-документацию:

- HTTPS/reverse proxy;
- BotFather checklist для Mini App;
- menu button setup;
- fallback при HTTP/local;
- smoke checklist сайта и Mini App.

## 12. Безопасность и приватность

Требования:

- доверять только серверной валидации `initData`;
- не хранить raw `initData` в логах;
- не логировать пользовательские заметки, названия партий, кормов и хозяйств как
  analytics payload;
- activity analytics оставлять metadata-only;
- web-ключи хранить только в хешированном виде, когда появятся персональные
  ключи;
- ошибки авторизации не раскрывают наличие пользователя;
- админский UI и admin API не доступны через пользовательскую навигацию;
- экспорт данных требует отдельного ТЗ на формат и подтверждение.

## 13. Наблюдаемость

Логировать без чувствительных параметров:

- запуск web-сервиса;
- открытие сайта;
- открытие Mini App через Telegram `initData`;
- открытие через web login token;
- auth failed;
- api error;
- write operation failed.

В админской аналитике показывать агрегаты, но не тексты пользовательских данных.

## 14. Критерии приемки MVP Mini App

1. При `WEB_PUBLIC_URL=https://...` пользователь в Telegram видит выбор
   `Открыть сайт` / `Открыть Mini App`.
2. `Открыть Mini App` запускается через Telegram `web_app` кнопку только на HTTPS.
3. Backend принимает `X-Telegram-Init-Data`, валидирует подпись и создает или
   обновляет пользователя.
4. При невалидном или просроченном `initData` пользовательские endpoints
   возвращают `401`.
5. Вне Telegram сайт продолжает открываться через безопасный web-login механизм.
6. Обычный пользователь не видит admin UI и не может вызвать admin-сценарии из
   Mini App.
7. Разделы `Сводка`, `Корма`, `Смесь`, `Поголовье`, `Яйца`, `Инкубация`,
   `Настройки` проходят smoke-тест на мобильной ширине.
8. UI корректно работает в светлой и темной теме Telegram.
9. Нет browser-native `prompt`, `alert`, `confirm`.
10. Запись из сайта или Mini App отражается в Telegram-боте и наоборот.
11. `tests/test_web_app.py` и общий test suite проходят.
12. Deployment docs содержат шаги HTTPS, BotFather/menu button и fallback.

## 16. Проверки

Базовые автоматические проверки:

```powershell
python -B -m pytest -q
python -B scripts\smoke_start.py
python -B scripts\web_app.py
```

`scripts\web_app.py` является ручной проверкой запуска web-сервиса. Для CI
нужен отдельный dry-run или health-check сценарий, чтобы не оставлять серверный
процесс висеть в тестовом pipeline.

После добавления Mini App нужны дополнительные проверки:

```powershell
python -B -m pytest -q tests\test_web_app.py
python -B -m pytest -q tests
```

Ручная приемка:

- открыть сайт по HTTPS;
- открыть Mini App из Telegram;
- проверить светлую и темную тему;
- добавить покупку корма;
- создать замес;
- добавить сбор яиц;
- создать или изменить группу поголовья;
- открыть Telegram-бот и убедиться, что изменения видны;
- сделать обратную правку в Telegram-боте и убедиться, что она видна на сайте и
  в Mini App.
