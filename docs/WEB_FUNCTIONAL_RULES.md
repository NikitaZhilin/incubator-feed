# Функциональные правила сайта и Mini App

Документ фиксирует текущую продуктовую логику: какие действия нужны в сайте и
Mini App, что именно записывается в БД и как считаются показатели. Авторизация,
production hardening и публичная безопасность вынесены в
`docs/FINAL_HARDENING_BACKLOG.md`.

## 1. Общий принцип

Telegram-бот, сайт и Mini App должны работать с одними и теми же сущностями:

- пользователь Telegram;
- инкубационные партии;
- складские позиции;
- складские операции;
- замесы;
- группы поголовья;
- стада;
- назначения смеси стаду;
- записи сбора яиц;
- исключения несушек;
- настройки хозяйства.

Бизнес-логика должна оставаться в `app/services/*`. Web/Mini App слой только
собирает форму, вызывает service и показывает результат.

## 2. Склад

### Покупка

Пользователь вводит:

- название позиции;
- тип: ингредиент, готовая смесь, готовый корм, другое;
- количество;
- заметку.

Запись:

- позиция находится или создается в `stock_items`;
- операция записывается в `stock_transactions` с типом `purchase`;
- `balance_after_kg` равен текущему расчетному остатку плюс покупка.

Расчет:

- текущий остаток считается от последней операции;
- после последней операции вычитается расчетный расход по активным назначениям;
- `days_left = floor(remaining_kg / daily_usage_kg)`;
- `ends_at = today + days_left`;
- `purchase_by = today + max(days_left - 7, 0)`;
- если расхода нет, `days_left`, `ends_at` и `purchase_by` не считаются;
- в web/Mini App показывается последняя складская операция по позиции: покупка, замес, списание или ручная корректировка.

### Нормы расхода корма

Базовые нормы для расчета по стадам:

- курица/несушка: `120 г/день`;
- петух: `150 г/день`;
- взрослая смешанная птица: `120 г/день`.

Нормы для цыплят считаются по возрасту от даты вывода:

- 0-7 день: `15 г/день`;
- 8-14 день: `25 г/день`;
- 15-28 день: `45 г/день`;
- 29-56 день: `70 г/день`;
- 57-84 день: `90 г/день`;
- с 85 дня: `110 г/день`.

Формула назначения смеси стаду:

`daily_usage_kg = birds_count * daily_norm_g * share_percent / 100 * (1 + reserve_percent / 100) / 1000`

Для цыплят отдельный расход цыплячьего корма останавливается с даты подсадки. Если цыплята включены в стадо и для стада назначена смесь, расход по ним считается в составе стада по возрастной норме.

### Корректировка фактического остатка

Пользователь выбирает позицию склада и вводит фактический остаток.

Запись:

- старая покупка или замес не удаляются из истории;
- создается новая операция `manual_adjustment` в `stock_transactions`;
- `amount_kg` и `balance_after_kg` равны введенному фактическому остатку;
- заметка сохраняется как причина корректировки.

Расчет после корректировки:

- дальнейший остаток считается от этой операции;
- расход после времени корректировки снова вычитается по активным назначениям;
- ошибочную покупку в тестовом web/Mini App сценарии исправляем именно корректировкой, а не физическим удалением.

## 3. Замес

### Расчет замеса

Пользователь выбирает:

- число замесов;
- зерновую основу, если доступно несколько вариантов.

Расчет:

- рецепт берется из `app.services.feed_recipes`;
- требуемый вес ингредиента = `parts * density_kg_per_l * mix_count`;
- складская позиция ищется по названию и алиасам;
- доступный остаток берется из расчетного остатка склада;
- замес разрешен только если у каждого ингредиента `missing_kg <= 0`;
- максимальное число замесов определяется самым ограничивающим ингредиентом.

### Создание замеса

Запись:

- создается запись `mix_productions`;
- по каждому ингредиенту создается `mix_production_items`;
- по каждому ингредиенту создается операция `mix_input` с отрицательным весом;
- для готовой смеси создается или находится `stock_item`;
- создается операция `mix_output` с полученным весом готовой смеси.

Правило UX:

- перед списанием ингредиентов обязательно показывать экран подтверждения.

## 4. Поголовье

### Создание группы

Пользователь вводит:

- название;
- количество птиц;
- вид;
- тип: взрослая группа или цыплята;
- роль: несушки, петухи, смешанная взрослая группа, цыплята;
- дату вывода;
- дату подсадки;
- процент запаса.

Правила:

- название не пустое;
- количество больше нуля;
- для цыплят обязательна дата вывода;
- дата подсадки не может быть раньше даты вывода;
- взрослой группе нельзя поставить роль `цыплята`;
- для цыплят роль всегда `цыплята`.

Запись:

- создается строка в группах поголовья через `FeedService.create_bird_group`.

### Редактирование группы

Можно менять:

- название;
- количество;
- роль взрослой группы;
- дату вывода;
- дату подсадки;
- процент запаса.

Расчеты после изменения должны обновляться на сводке, складе, яйцах и стадах.

## 5. Стада

### Создание стада

Пользователь вводит:

- название стада;
- одну или несколько групп поголовья.

Правила:

- название не пустое;
- в стаде должна быть хотя бы одна активная группа;
- каждая выбранная группа должна принадлежать текущему пользователю.
- у пользователя не может быть двух активных стад с одинаковым названием без
  учета регистра и пробелов по краям;
- если стадо уже существует, нужно редактировать его состав, а не создавать
  дубль.

Запись:

- создается `flock`;
- для каждой выбранной группы создается связь `flock_member`.

### Редактирование стада

Можно менять:

- название;
- состав групп.

Правило:

- при изменении состава удаляются связи с убранными группами и добавляются связи
  с новыми.
- переименование не может пересечься с названием другого активного стада этого
  же пользователя.

## 6. Назначение смеси стаду

Пользователь выбирает:

- стадо;
- готовую смесь со склада.

Правила:

- стадо должно быть активным;
- складская позиция должна существовать;
- стаду можно назначить только позицию типа `finished_mix`;
- сумма долей рациона не может быть больше 100%.

Запись:

- создается `flock_feed_assignment`;
- старые назначения на группы этого стада деактивируются, чтобы не было двойного
  расхода.

Расчет расхода:

- несушка: `daily_per_hen_g`;
- петух: `daily_per_rooster_g`;
- взрослая смешанная группа: `daily_per_adult_g`;
- цыплята: расход зависит от возраста;
- применяется доля рациона `share_percent`;
- применяется запас `reserve_percent`.

## 7. Яйца

### Запись сбора

Пользователь вводит:

- дату сбора;
- количество яиц;
- заметку.

Запись:

- создается запись в `egg_entries`;
- фиксируется количество яиц;
- фиксируется число несушек на дату записи;
- фиксируется число временно исключенных несушек;
- заметка сохраняется как часть записи.

### Редактирование и удаление

Пользователь может исправить:

- дату сбора;
- количество яиц;
- заметку.

При редактировании пересчитываются:

- общее число несушек на новую дату;
- число исключенных несушек на новую дату;
- число активных несушек.

Ошибочную запись можно удалить. Для удаления требуется явное подтверждение в
форме. После удаления статистика пересчитывается без этой строки.

Расчет несушек:

- считаются активные группы поголовья;
- учитываются только взрослые группы с ролью `hens`;
- временные исключения уменьшают число активных несушек;
- исключений не может быть больше общего числа несушек.

### Статистика

Считается:

- яйца за сегодня;
- сумма за 7 дней;
- сумма за 30 дней;
- среднее за 7 дней только по дням с записями;
- среднее за 30 дней только по дням с записями;
- яйца на активную несушку;
- прогноз на неделю = среднее за 7 дней * 7;
- погодная поправка применяется отдельно и показывается как ориентировочная.

## 8. Временно не несутся

Пользователь вводит:

- количество кур;
- причину;
- дату начала;
- ожидаемую дату окончания.

Правила:

- количество больше нуля;
- причина из разрешенного списка;
- если несушки заведены, количество не может быть больше общего числа несушек;
- дата окончания не может быть раньше даты начала.

Запись:

- создается активное исключение;
- при завершении проставляется дата окончания.

## 9. Погода

Пользователь вводит город.

Запись:

- город сохраняется в настройках погоды пользователя;
- при доступном weather client сохраняются координаты и provider.

Расчет:

- сохраненная погода используется для ориентировочной поправки к прогнозу яиц;
- если данных нет, поправка 0 и показывается объяснение;
- сильный холод, жара и осадки могут снижать прогноз.

## 10. Инкубация

Сайт и Mini App должны показывать:

- активные партии;
- завершенные партии;
- вид птицы;
- количество яиц;
- дату закладки;
- текущий день;
- этап;
- ожидаемую дату вывода;
- рекомендации на текущий день;
- результат завершенных партий.

На текущем web-этапе уже есть изменение:

- создать партию;
- завершить вывод;
- вернуть завершенную партию в активные.

Для полного parity нужно:

- отредактировать активную партию: вид птицы, количество яиц, дату закладки, название и заметку.

## 11. Настройки

Сайт и Mini App должны показывать:

- Telegram ID;
- название хозяйства;
- часовой пояс;
- время уведомлений;
- включенные разделы;
- город погоды;
- версию;
- канал релиза;
- changelog/GitHub.

На текущем web-этапе уже есть изменение:

- города погоды;
- включения/выключения разделов.

Для полного parity нужны:

- изменение названия хозяйства;
- изменение часового пояса;
- изменение времени уведомлений.

## 12. Ежедневная сводка Telegram

Worker `reminder_runner` один раз в день после 12:00 по часовому поясу пользователя отправляет активному пользователю сводку хозяйства. Повторная автоматическая отправка за тот же день блокируется через `notification_log` с типом `daily_summary`. Пользователь может вызвать ту же сводку вручную кнопкой `📊 Посмотреть сводку` в главном меню.

Сводка содержит:

- напоминание собрать яйца и записать результат;
- напоминание проверить и заменить воду;
- напоминание дать корм;
- статистику яиц за сегодня и за 7 дней;
- дату последнего замеса готовой смеси;
- остаток готовой смеси;
- расчетный расход готовой смеси в день;
- срок, на сколько хватит смеси;
- дату, когда смесь закончилась по расчету, и сколько дней назад это было, если расчетный остаток уже нулевой;
- критическое предупреждение, если смеси осталось на 2 дня или меньше;
- подсказку, сколько замесов можно сделать из текущих складских ингредиентов, либо каких ингредиентов не хватает.

Под сводкой показываются inline-кнопки:

- `Добавить яйца` -> `eggs:add`;
- `Переход в раздел корма` -> `feeds:menu`;
- `Выйти в меню` -> `menu:home`.

## 13. Приоритет текущей тестовой разработки

Уже доведено до удобного web-тестирования:

1. Инкубация: создание и завершение партии из web/Mini App.
2. Склад: фактический остаток и отмена ошибочной покупки через
   корректирующую операцию.
3. Стада: защита от дублей активного стада с тем же названием и подсказка
   редактировать существующее стадо.

Следующие функциональные блоки:

1. Поголовье: архивирование группы и понятная смена роли.
2. Стада: архивирование стада и расширенное переназначение смеси.
3. Настройки: название хозяйства, часовой пояс, время уведомлений.

Авторизация, персональные web-ключи и Mini App production hardening выполняются
после проверки этих сценариев администратором.
